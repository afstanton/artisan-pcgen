# artisan-pcgen Design

## Purpose

`artisan-pcgen` is a lossless round-trip parser and emitter for PCGen `.lst`,
`.pcc`, and `.pcg` data files. It converts raw PCGen text into structured
`artisan-core` `Entity` values and back, preserving every token exactly.

`.lst` and `.pcc` are line-oriented list/catalog sources; `.pcg` is the
character-sheet format and is part of the same compatibility contract.

---

## Pipeline

```
PCGen text (.lst / .pcc / .pcg)
  │
  ▼ parse_*()              (LALRPOP + Logos lexer)
Parsed structures (line/section scoped)
  │
  ▼ project()              (src/analysis/fields.rs + format adapters)
Entity { name, attributes, effects, prerequisites }
  │
  ▼ emit_*()               (src/emit.rs + format adapters)
PCGen text (.lst / .pcc / .pcg)  ← identical semantics to input
```

The current implementation focus is line-oriented `.lst`/`.pcc`. `.pcg`
parsing/emission is a required extension of the same model.

Each stage is driven by an `EntitySchema` registered in
`src/schema/mod.rs::ALL_SCHEMAS`.

---

## Schema-Driven Architecture

Every PCGen entity type is described by an `EntitySchema`:

- **`head_token`** / **`head_format`** — how the identifying head clause is
  parsed and emitted.
- **`tokens: &[TokenDef]`** — the ordered list of clauses the entity supports.
  Each `TokenDef` carries:
  - `key` — the PCGen token string (e.g. `"SOURCEPAGE"`).
  - `grammar` — value shape (`Text`, `PipeList`, `YesNo`, `PipePositional`, …).
  - `cardinality` — `Once` or `Repeatable`.
  - `artisan_mapping` — how the token is stored in the `Entity`:
    - `Attribute(field)` — stored in `entity.attributes[field]`.
    - `Effect` — stored as an entry in `entity.effects`.
    - `Prerequisite` — stored in `entity.prerequisites`.
    - `EntityName` — maps to the entity name itself (head token).
- **`globals`** — shared token groups (BONUS, TYPE, FACT, DESC, …) reused
  across all entity types via `GlobalGroup`.

Projection (`fields.rs`) translates the flat `ParsedClause` stream into the
typed artisan representation.  Emission (`emit.rs`) is the exact inverse:
schema token order is authoritative, so the emitter always produces canonical
output regardless of the original token order in the source file.

## Token Completeness Goal

**Every PCGen token that `artisan-pcgen` lexes and parses shall also be
generatable** — given a properly populated `Entity`, the emitter must produce
correct output for that token.

Concretely, this means every recognised token must have:

1. A `TokenDef` in the appropriate `EntitySchema` with an
   `ArtisanMapping` of `Attribute`, `Effect`, or `Prerequisite` (never
   `None`).
2. A corresponding arm in `src/analysis/fields.rs` that projects the parsed
   value into the mapped artisan field.

The **only** exception is **deprecated tokens** — PCGen tokens superseded by
a replacement.  These should project onto the *replacement* token's artisan
field so the emitter silently upgrades them.

There is no "non-representational" exception.  The emission target is
**PCGen compatibility**: files produced by `artisan-pcgen` must be valid,
loadable PCGen data.  This means every token PCGen recognises must survive the
round-trip even if Artisan does not interpret its semantic meaning.

### Value Semantics Policy

Token values are **semantic by default**. If a token value has functional
effect in PCGen (loader behavior, UI behavior, rule behavior, filtering,
selection, visibility, legality, etc.), the value must not be treated as an
opaque raw blob. It should be projected into explicit artisan fields with a
stable structure that can be transformed into other output formats.

Raw text storage is allowed only for content that is intentionally
non-functional, such as comments or equivalent annotation text. Those may be
preserved verbatim across formats.

For compatibility-oriented tokens that Artisan does not otherwise consume, the
minimum acceptable behavior is still to model and preserve them as explicit
attributes (and evolve structure over time), not to suppress them with
`ArtisanMapping::None`.  Examples:

- `FORWARDREF` — a loader cross-reference hint PCGen requires for some data
  sets.  Artisan doesn't resolve references itself, but it must preserve and
  emit this functional directive for PCGen.
- `HIDETYPE` — a PCGen UI display hint with no artisan equivalent.  Store as
  structured attributes and re-emit for compatibility.
- `LICENSE`, `INFOTEXT`, `URL` — legal and informational metadata.  No
  core artisan semantics, but they belong in the output file and should be
  represented as explicit attributes.

`ArtisanMapping::None` therefore has no legitimate permanent use in steady
state. Any token currently assigned `None` is backlog.

Progress toward this goal is tracked by `TOKEN_INVENTORY.txt`, regenerated by
`cargo run --bin token_inventory`.  The target metrics are:

| Metric | Target |
|---|---|
| Semantically interpreted but not fully-structured | 0 |
| Unhandled tokens | 0 |

---

## Token Classification (`TOKEN_INVENTORY.txt`)

| Classification | Meaning |
|---|---|
| **Fully-structured** | Has a `TokenDef` *and* a `fields.rs` projection; full round-trip guaranteed. |
| **Semantically interpreted** | Has a `fields.rs` projection but no `TokenDef` (interim state). |
| **Policy-only (transitional)** | Handled by `token_policy.rs` today; target is to migrate to schema + projection or remove if deprecated. |
| **Unhandled** | Lexed as an opaque `KEY:VALUE` pair; no projection, no emission. |

---

## Round-Trip Guarantee

The `tests/semantic_roundtrip.rs` integration test enforces the guarantee for
line-oriented data: parse fixture `.lst`/`.pcc` content, project to entities,
emit back to text, assert byte-identical output. Every token in
`tests/fixtures/pcgen/roundtrip_structured_token_coverage.lst` must pass this
test. New tokens are added to the fixture as they are promoted to
fully-structured.

Equivalent round-trip fixtures and invariants are required for `.pcg`
character files as `.pcg` support is implemented.
